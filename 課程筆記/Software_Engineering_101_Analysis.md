## Requirements Engineering

歡迎來到需求工程（Requirements Engineering, RE）的核心世界。在軟體工程與系統開發中，需求工程是決定專案成敗的關鍵起點。它的任務是將用戶腦中模糊、抽象的「想法」，轉化為工程師能夠精準實作的「系統規格」。

以下為您由淺入深、系統性地拆解這些核心概念：

## 1. 需求定義 (Requirements Definition)

需求定義是需求工程的初期產物。它是用**客戶與使用者看得懂的自然語言（非技術性語言）**，搭配簡單的圖表，來描述系統預期需要提供什麼服務，以及它必須在什麼限制下運作。

- **核心目的：** 建立開發團隊與客戶（利益相關者）之間的共識橋樑，確保雙方對「要做什麼」有相同的理解。
    
- **主要內容：** 通常包含高階的業務目標（Business Goals）、使用者故事（User Stories）以及系統邊界。
    

## 2. 功能需求 vs. 非功能需求

在收集與分析需求時，我們會將其明確區分為兩大類：

### 功能需求 (Functional Requirements)

- **定義：** 系統「必須做什麼」。描述系統必須提供的具體功能、服務、反應或資料處理邏輯。
    
- **白話範例：** * 「當使用者按下『結帳』按鈕時，系統必須計算購物車總金額並跳轉至付款頁面。」
    
    - 「系統必須能自動生成月度銷售報表，並匯出為 PDF 格式。」
        
- **檢驗標準：** 系統有沒有這個功能？（非黑即白）。
    

### 非功能需求 (Non-Functional Requirements)

- **定義：** 系統「必須如何運行」。描述系統的品質屬性、效能限制、安全性或使用者體驗等。它不直接參與業務運作，但決定了系統「好不好用」、「安不安全」。
    
- **白話範例：**
    
    - **效能（Performance）：** 「系統在同時在線人數 10,000 人時，網頁響應時間必須小於 2 秒。」
        
    - **安全性（Security）：** 「所有使用者密碼在寫入資料庫前，必須經過 SHA-256 加密。」
        
    - **可靠性（Availability）：** 「系統必須達到 99.9% 的全年可用性（SLA）。」
        
- **檢驗標準：** 通常是程度問題，需要量化指標來衡量。
    

## 3. 規格書 (Specifications)

規格書（Software Requirements Specification, SRS）則是更進一步、**寫給工程師與測試人員看的技術性文件**。

- **轉化過程：** 它將「需求定義」中那些抽象的客戶語言，精確地翻譯成軟體系統的具體行為、輸入輸出、介面定義與資料結構。
    
- **特點：** 規格書必須具備**可驗證性（Verifiable）**、**無歧義性（Unambiguous）**與**完整性**。測試人員會直接根據規格書來編寫測試案例（Test Cases）。
    

## 4. WRSPM 模型

**WRSPM 模型**（由 Michael Jackson 與 Pamela Zave 提出）是需求工程中非常經典且嚴謹的數學與邏輯框架。它將現實世界與軟體系統之間的關係，切分為五個核心概念，用來確保需求的正確性與完備性。

這五個字母分別代表：

- **W (World / Domain Knowledge) — 領域知識：** 現實世界中**本來就存在的客觀事實、規則或物理限制**（不管你有沒有做這個系統，它都存在）。
    
    - _範例：_ 「火車如果緊急煞車，因為慣性，需要 200 公尺的緩衝距離才會完全停下。」
        
- **R (Requirements) — 需求：** 客戶**希望**在現實世界中達成的目標或效果（通常存在於現實環境中，而非系統內部）。
    
    - _範例：_ 「系統必須確保兩輛火車絕對不會在同一條軌道上相撞。」
        
- **S (Specification) — 規格：** 系統與現實世界交界處的**硬體/軟體介面行為**。這是開發團隊唯一能直接控制並用來改變世界的依據。
    
    - _範例：_ 「當感測器偵測到前方 500 公尺有另一輛火車時，系統必須對煞車控制器發送高電位訊號。」
        
- **P (Program) — 程式：** 工程師用 C++、Python 等語言寫出來的**核心原始碼**。
    
- **M (Machine) — 機器/硬體：** 執行程式的**硬體平台、作業系統或底層硬體環境**（如樹莓派、伺服器）。
    

### WRSPM 的邏輯核心（公式）

這個模型的精髓在於它的邏輯推導關係，如果你的設計是正確的，以下兩個公式必須成立：

1. M,P⊢S
    
    - **意涵：** 你的程式（P）跑在機器硬體（M）上，必須能夠完全實現規格書（S）所定義的行為。
        
2. W,S⊢R
    
    - **意涵（至關重要）：** 結合了現實世界的物理定律/事實（W），再加上你寫的規格行為（S），必須能夠**百分之百滿足客戶的需求（R）**。
        

> **白話反思：** 很多專案失敗，不是因為程式（P）寫不好，而是因為**忽略了現實世界 Fact (W)**。 例如：你寫的規格（S）是在距離 100 公尺時煞車，程式（P）也完美執行了。但因為你不知道火車煞車需要 200 公尺（W），導致最後火車還是相撞了（需求 R 失敗）。WRSPM 模型就是為了逼工程師在動手前，把這些邊界條件想得清清楚楚。

## Software Architecture

歡迎來到軟體架構（Software Architecture）的核心世界。軟體架構是大型系統的設計藍圖，它決定了系統的組件如何劃分、這些組件如何通訊，以及系統如何滿足效能、可擴展性與維護性等非功能需求。

以下為您系統性地拆解四種最經典的軟體架構模式（Architectural Patterns）：

## 1. 管線與過濾器 (Pipe & Filter)

Pipe & Filter 是一種將資料流（Data Stream）處理分解為多個獨立、可重複使用步驟的架構模式。這非常適合用於資料處理、影像處理或編譯器設計。

- **組件拆解：**
    
    - **Filter（過濾器）：** 負責執行具體的資料轉換或處理任務。它只專注於輸入資料，處理完後輸出，彼此之間**完全不知道**其他 Filter 的存在（高度解耦）。
        
    - **Pipe（管線）：** 負責將一個 Filter 的輸出資料流，傳送到下一個 Filter 的輸入端（負責緩衝與傳輸）。
        
- **白話範例：** Linux 的命令列管線機制。例如：`cat log.txt | grep "ERROR" | awk '{print $3}'`。
    
    - `cat`、`grep`、`awk` 都是獨立的 Filter，而 `|` 就是 Pipe。
        
- **優缺點：** 很容易透過調整 Filter 的順序來重組系統，且支援平行運算；但由於每個 Filter 都要重覆進行資料的序列化與反序列化，可能會帶來效能開銷。
    

## 2. 客戶端-伺服器 (Client-Server)

Client-Server 是分散式系統中最廣泛使用的架構模式。它將系統劃分出兩個不同的角色，通常透過網路進行通訊。

- **組件拆解：**
    
    - **Client（客戶端）：** 服務的請求者與使用者介面的呈現者。它向伺服器發送請求，並將回傳的結果展示給用戶。
        
    - **Server（伺服器）：** 服務的提供者與資源的管理者的中心。它會持續監聽來自客戶端的請求，處理業務邏輯，並存取資料庫以回傳結果。
        
- **白話範例：** 網頁瀏覽器（Client，如 Chrome）與網站伺服器（Server，如 Google 的後端）。
    
- **優缺點：** 資源集中管理，資料一致性高，且客戶端可以非常輕量化；但缺點是伺服器容易成為單點故障（Single Point of Failure, SPOF）與效能瓶頸。
    

## 3. 主-從架構 (Master-Slave / Leader-Follower)

Master-Slave 架構專門用於提升系統的**運算效能、可靠性與負載平衡**。它引入了不對稱的控制權限。

- **組件拆解：**
    
    - **Master（主設備/領導者）：** 負責統籌調度、分配工作與接收寫入指令。它是整體的決策中心。
        
    - **Slave（從設備/跟隨者）：** 負責執行 Master 分配下來的具體運算任務，或是複製 Master 的狀態以提供唯讀服務。
        
- **實務應用情境：**
    
    - **資料庫讀寫分離：** Master 資料庫負責處理寫入（Write）指令，並同步資料給多台 Slave 資料庫；多台 Slave 資料庫則專職負責處理大量的讀取（Read）請求，大幅提升資料庫併發效能。
        
    - **平行運算（如 MapReduce）：** Master 將巨大的資料切分成小塊，分發給數十台 Slave 電腦同步運算，最後再由 Master 收集結果。
        
- **優缺點：** 容錯性高（若 Slave 掛掉，Master 可以找別台頂替；若 Master 掛掉，可從 Slave 中推選新 Master），且水平擴展性好。
    

## 4. 分層架構 (Layered Architecture / N-Tier Architecture)

分層架構是最普遍、最符合人類思維的傳統架構模式。它將系統水平切分為多個層級，每一層都擁有專屬的職責，且**上層只能調用直接下層提供的服務**（關注點分離）。

一個標準的三層或四層 Web 應用架構通常如下：

1. **表現層 (Presentation Layer / UI)：** 負責與使用者互動、呈現網頁、接收輸入。
    
2. **業務邏輯層 (Business Logic Layer / Service)：** 系統的核心大腦，處理具體的商務邏輯與演算法。
    
3. **資料存取層 (Data Access Layer / DAO)：** 負責撰寫 SQL 語句，作為系統與資料庫之間的橋樑。
    
4. **資料庫層 (Database Layer)：** 實際儲存數據的底層實體（如 MySQL、Oracle）。
    

- **優缺點：** * **極高的可維護性：** 如果今天決定把資料庫從 MySQL 換成 PostgreSQL，你只需要修改「資料存取層」，上層的「業務邏輯層」和「表現層」程式碼完全不需要更動。
    
    - **缺點（過度轉發）：** 有時候一個簡單的讀取需求，資料必須被迫層層傳遞（表現層 $\rightarrow$ 業務層 $\rightarrow$ 資料層），會產生些微的效能損耗。

## Software Design

歡迎來到軟體設計（Software Design）的核心領域。如果說軟體架構（Architecture）是決定蓋大樓還是蓋橋樑的「宏觀藍圖」，那麼軟體設計與這些核心原則，就是決定磚塊與結構如何組織、如何確保系統在未來「容易修改、不會牽一髮而動全身」的「微觀工藝」。

以下為您系統性地拆解這四個最經典的軟體設計基石：

## 1. 模組化 (Modularity)

模組化是一種「分而治之（Divide and Conquer）」的設計思想。它是指將一個巨大、複雜的軟體系統，拆分成數個互相獨立、職責單一且可單獨開發與測試的「模組（Modules）」（在程式中可以是 Package、Component、Class 或 Function）。

- **核心目的：** 降低系統的複雜度。人類的大腦無法同時理解百萬行的程式碼，但如果拆成 100 個各自獨立的模組，我們就能逐一攻破。
    
- **白話範例：** 樂高積木（Lego）或汽車零件。引擎、輪胎、變速箱都是獨立模組，它們可以被單獨製造、測試，最後拼裝在一起。
    

## 2. 封裝 (Encapsulation)

封裝是物件導向程式設計（OOP）的核心特性之一，也是實踐模組化的具體手段。它是指**將資料（Data / Attributes）與處理這些資料的方法（Methods / Functions）包裝在同一個單元內，並隱藏內部的實作細節，僅暴露必要的公開介面（Interface）**。

- **核心目的：** 資訊隱藏（Information Hiding）與保護資料。外部程式碼不能隨意修改模組內部的狀態，只能透過允許的通道（如 Getter / Setter 或 API）來互動。
    
- **白話範例：** 自動販賣機。你只需要投錢（Input）、按下按鈕（Interface），就能拿到飲料（Output）。你不需要（也沒權限）知道販賣機內部齒輪是怎麼轉動、零錢是怎麼分類的。
    

## 3. 耦合度 (Coupling) — 越低越好 (Low Coupling)

耦合度用來衡量**不同模組之間相互關聯、相互依賴的緊密程度**。

- **緊密耦合（Tight Coupling - 壞設計）：** 模組 A 內部修改了某行程式碼，結果導致完全不相關的模組 B 和模組 C 崩潰編譯失敗。這會造成嚴重的「連鎖反應」，讓系統變得極難維護。
    
- **鬆散耦合（Loose Coupling - 好設計）：** 模組之間僅透過明確定義的**介面**進行極少的通訊。模組 A 內部不論怎麼大改、優化重構，只要對外的公開介面（輸入與輸出規格）不變，其他模組完全不受影響。
    

## 4. 內聚度 (Cohesion) — 越高越好 (High Cohesion)

內聚度用來衡量**單一模組內部各個元素（程式碼、方法、變數）之間，為了達成同一個目標而聚集在一起的緊密程度**（專注度）。

- **低內聚（Low Cohesion - 壞設計）：** 一個模組裡塞滿了各式各樣雜七雜八的功能。例如一個叫 `Util` 的類別，裡面既處理「資料庫連線」、又處理「字串格式化」、還順便處理「傳送 Email」。這就像一個什麼都賣卻什麼都不精的雜貨店，極難重複使用與測試。
    
- **高內聚（High Cohesion - 好設計）：** 一個模組**只專注做好一件事情**（符合單一職責原則 SRP）。例如一個 `EmailSender` 模組，裡面所有的程式碼都只為了「寄信與驗證信件格式」服務。
    

## 💡 總結：軟體設計的終極聖經

不論是物件導向設計、微服務架構，還是寫簡單的函數，軟體設計的終極目標永遠是這八個字：

> **「高內聚，低耦合 (High Cohesion, Low Coupling)」**

- 我們透過**模組化**與**封裝**，把相關的資料與邏輯關在一起，確保模組內部**極度專注（高內聚）**。
    
- 同時限制模組與外部的管道，只走標準介面，確保模組與模組之間**劃清界線、互不牽絆（低耦合）**。 這樣設計出來的軟體，才能在不斷變更的需求下，維持極高的生命力與擴展性。

## Deployment

歡迎來到軟體部署與架構決策（Deployment & Strategic Decisions）的核心領域。當系統完成設計與開發後，如何將其推向生產環境（Production），以及在開發初期如何選擇軟體來源，是決定企業營運成本與系統穩定性的關鍵。

以下為您系統性地拆解這四個關鍵技術與決策節點：

## 1. 自建 vs. 購買 (Buy vs. Build)

這是所有技術主管（CTO/CIO）在專案啟動時面臨的第一個**戰略性決策**。當企業需要某個軟體功能時（例如：CRM 系統、通訊中間件、或 AI 客服系統），應該自己寫程式開發（Build），還是直接買市面上的現成產品（Buy）？

這通常需要從**核心競爭力**與總體擁有成本（TCO）來評估：

|維度|自行開發 (Build)|直接購買 (Buy / SaaS)|
|---|---|---|
|**商務優勢**|擁有完全的控制權，能建立企業的**核心技術壁壘**與獨特創新。|能夠**快速推向市場（Time to Market）**，直接使用業界成熟的最佳實踐。|
|**客製化程度**|**極高**。完全量身打造，能 100% 符合企業現有的特殊業務流程。|**較低**。必須適應供應商的架構，或付出高昂的二次開發代價。|
|**初期成本**|**極高**。需要投入大量的人力（研發團隊）與時間成本。|**較低**。通常是按年/按使用量訂閱（OpEx），初期資金壓力小。|
|**長期維護**|企業必須自己承擔程式碼修補、升級與資安漏洞修復（如 ISO 37001 合規維護）。|由供應商負責維護與升級，企業只需專注於使用。|

> **決策黃金法則：**
> 
> - 如果這個功能是企業的**核心獲利能力**（例如：高鐵車載通訊箱的 localization 核心韌體），應選擇 **Build**。
>     
> - 如果這個功能只是**支援性商品**（例如：企業內部的差勤打卡系統、通用型 Email 發送服務），應優先選擇 **Buy**。
>     

## 2. 部署規劃 (Deployment Planning)

部署規劃是指將軟體從測試環境安全移轉到生產環境的**專案管理與技術準備過程**。一份嚴謹的部署計畫是防止「上線即崩潰」的唯一防線。

### 部署計畫的核心要素：

- **環境對齊（Environment Alignment）：** 確保測試環境（Staging）與生產環境（Production）的硬體規格、作業系統版本、資料庫配置完全一致。現代化部署通常依賴 **Docker 容器化** 與 **IaC（基礎設施即代碼，如 Terraform）** 來確保環境一致性。
    
- **資料庫遷移計畫（Database Migration）：** 如果新版本修改了資料庫欄位結構（Schema），必須規劃好如何在不中斷業務的情況下執行 SQL 遷移腳本，並確保新舊程式碼在過渡期都能讀寫資料（相容性設計）。
    
- **動態黑名單與權限查核：** 上線前的最終檢查，包含第三方 API 金鑰、敏感政府專案（Sensitive Government Project）的資安稽核等。
    

## 3. 部署策略 (Deployment Strategy)

部署並不是簡單地把新程式蓋過去，現代雲端架構通常採用以下兩種主流的無中斷部署策略，以降低上線風險：

### A. 藍綠部署 (Blue-Green Deployment)

系統同時維持兩套完全相同的實體環境：

- **藍環境 (Blue)：** 目前正在線上面對真實用戶的**舊版本**環境。
    
- **綠環境 (Green)：** 部署了**新版本**的待命環境。
    
- **運作機制：** 在綠環境完成所有內部測試後，透過負載平衡器（Load Balancer）或路由器，**瞬間（0.1 秒）將流量從藍色切換到綠色**。如果新版本有問題，可以立刻一鍵切換回藍色。
    

### B. 金絲雀部署 (Canary Deployment)

不一次性切換所有流量，而是採取**漸進式升級**：

- **運作機制：** 先將 5% 的用戶流量導向部署了新版本的小型伺服器群（金絲雀），其餘 95% 留在舊版本。
    
- **監控觀察：** 觀察這 5% 用戶的系統錯誤率、CPU 負載與用戶回饋。若一切正常，再逐步將流量放大至 20%、50%，最終達到 100% 全面升級。這能將新版本出錯帶來的商譽損失控制在最小範圍。
    

## 4. 回滾策略 (Rollback Strategy)

在軟體工程中，墨菲定律永遠適用：「會出錯的事總會出錯。」**回滾（Rollback）是指當新版本軟體上線後出現嚴重故障時，將系統恢復到上一個穩定版本的緊急應變機制。**

**沒有回滾策略的部署計畫，不叫部署計畫，叫「冒險」。**

### 核心回滾實踐：

- **自動化健康檢查（Health Checks）：** 部署系統（如 Kubernetes）上線新版本後，會持續監測應用的探針。如果連續 5 分鐘重啟或回傳 500 錯誤，系統必須觸發**自動回滾**。
    
- **資料庫防禦（Data Preservation）：** 最難處理的回滾往往不是程式碼（程式碼只需退回舊版本版號即可），而是**資料**。如果在上線的 10 分鐘內，已經有用戶寫入了新欄位的資料，回滾時必須有機制將這些新資料導回舊結構，或是執行「逆向遷移腳本（Down Migration）」，避免資料遺失或損毀。
    
- **功能開關（Feature Toggles / Feature Flags）：** 這是現代高階回滾的技巧。將新功能用一個 `if (is_feature_active)` 的開關包起來。如果新功能上線後導致系統變慢，工程師不需要重新部署或回滾整個系統，只需在後台管理介面**關閉開關（Off）**，新功能便瞬間失效，系統立刻回到舊邏輯運行。

## Testing

歡迎來到軟體測試與品質保證（Software Testing & Quality Assurance）的核心領域。在軟體生命週期中，測試是確保系統符合預期規格、免於資安漏洞，並穩定交付給客戶的關鍵防線。

以下為您系統性地拆解從微觀到宏觀的測試核心概念與方法論：

## 1. 驗證與確認 (Verification & Validation / V&V)

這是軟體品質保證（QA）中最根本的哲學思考，兩者看似相似，但在軟體工程中有著完全不同的嚴謹定義：

- **驗證 (Verification) — 「我們有把系統做對嗎？」 (Are we building the product right?)**
    
    - **核心概念：** 檢查開發過程中的各個產出物，是否**符合前一個階段所設定的規格與限制**。這通常屬於「靜態檢查」。
        
    - **實務活動：** 程式碼審查（Code Review）、需求規格書檢查（如對齊 WRSPM 模型）、架構設計檢視。它不一定需要實際執行程式。
        
- **確認 (Validation) — 「我們有做對的系統嗎？」 (Are we building the right product?)**
    
    - **核心概念：** 檢查最終交付的軟體，是否**真正滿足客戶與使用者的實際需求與期望**。這屬於「動態檢查」。
        
    - **實務活動：** 執行軟體測試案例、用戶驗收測試（UAT）、效能壓測。它必須實際運行程式來觀察結果。
        

## 2. 測試層級：單元測試、整合測試與增量測試

軟體測試通常遵循經典的 **V 模型（V-Model）**，由內而外層層推進：

### A. 單元測試 (Unit Testing)

- **測試範疇：** 軟體中**最小的可測試單元**（通常是單一函數、方法或類別）。
    
- **核心特點：** 必須高度隔離。單元測試不應該依賴真實的資料庫、外部網路或硬體感測器。實務上會使用 **Mock（虛擬物件）** 來模擬外部元件。
    
- **實務效益：** 開發工程師在寫完程式碼當下即執行（如使用 Python 的 `unittest` 或 C++ 的 `GTest`），能在開發初期以極低成本抓到邏輯錯誤。
    

### B. 整合測試 (Integration Testing)

- **測試範疇：** 當多個單元單獨測試正常後，將它們**組合在一起，測試它們之間的介面與互動是否正確**。
    
- **白話範例：** 測試「購物車模組」呼叫「信用卡支付金流 API」時，資料傳遞是否正確。
    

### C. 增量測試 (Incremental Testing)

整合測試的一種主流策略。它不主張把所有模組一次全部拼裝起來測試（那叫大爆炸測試 Big Bang, 出了錯極難定位原因），而是**採取漸進式、做一塊拼一塊的方式進行測試**。

- **由上而下（Top-Down）：** 從系統的高階模組開始測試，底層尚未開發完的模組用 **Stub（樁模組/替代殼）** 代替。
    
- **由下而上（Bottom-Up）：** 從最底層的基礎函數、驅動程式開始測試，上層尚未完工的控制中心用 **Driver（驅動模組）** 來呼叫測試。
    

## 3. 測試視角：黑箱測試 vs. 白箱測試

根據測試人員「看不看得見程式碼內部結構」，測試方法分為兩大流派：

### A. 黑箱測試 (Black Box Testing / 功能性測試)

- **定義：** 測試人員**完全不知道、也不需要知道系統內部的程式碼是如何撰寫的**。
    
- **測試依據：** 完全依據**需求規格書（Specification）**。給予特定的輸入（Input），檢查輸出（Output）是否符合預期。
    
- **核心方法：**
    
    - **等價類劃分（Equivalence Partitioning）：** 將輸入資料區分為有效與無效的區間，從每個區間挑選代表值測試，避免窮舉測試。
        
    - **邊界值分析（Boundary Value Analysis）：** 經驗顯示錯誤最常發生在邊界（例如限制年齡 18-65 歲，則特意測試 17、18、65、66 歲）。
        

### B. 白箱測試 (White Box Testing / 結構性測試)

- **定義：** 測試人員（通常是工程師自己）**看得到原始碼，並根據程式碼內部的控制流、資料流與邏輯結構來設計測試案例**。
    
- **測試指標（代碼覆蓋率 Code Coverage）：**
    
    - **語句覆蓋（Statement Coverage）：** 確保每一行程式碼至少被執行過一次。
        
    - **分支覆蓋（Branch/Decision Coverage）：** 確保程式中每個 `if / else` 條件的「真」與「偽」路徑都有被測試到。
        
    - **路徑覆蓋（Path Coverage）：** 最嚴格，確保程式中所有可能的執行路徑組合都被跑過一遍（在複雜系統中容易遇到路徑爆炸）。
        

## 💡 系統整合思維：以真實機器人開發為例

將上述測試理論投入真實的專案（例如開發 AMR 自主移動機器人）：

1. **單元測試：** 工程師寫好輪圈運動學的 C++ 函數，輸入速度，驗證輸出的輪速脈衝是否正確（隔離實體馬達）。
    
2. **整合測試與增量測試：** 採用 **Bottom-Up 策略**。先測試 Arduino 驅動馬達（底層），接著將其與樹莓派的序列埠通訊整合，最後再把雷達、Nav2 導航模組一個個接上來，觀察空間代價地圖（Costmap）是否正確疊加。
    
3. **白箱測試：** 檢查導航狀態機（Behavior Tree）的程式碼分支，確保當雷達斷線（異常分支）時，系統能 100% 觸發安全停止（Safe Stop）邏輯。
    
4. **黑箱測試（確認 Validation）：** 站在終端客戶的角度，在 Rviz2 點擊目標點 B，測試機器人是否能安全繞過突然出現的障礙物並抵達終點，驗證產品是否真正好用、合規。

### Software Process Models
在軟體工程中，**軟體製程模型（Software Process Models）**（或稱軟體開發生命週期，SDLC）是引導開發團隊從最初的需求分析、設計、編碼、測試一路走向部署運行的**工作流程藍圖**。不同的模型適用於不同的專案規模、預算、時間限制與風險承受度。

以下為您由淺入深、系統性地拆解從傳統經典到現代演進的八大核心製程模型：

## 1. 瀑布模型 (Waterfall Model)

瀑布模型是最早、最經典的生命週期模型。它將開發過程視為如同瀑布一般**單向、不可逆**的線性序列。

- **工作流程：** 需求分析 $\rightarrow$ 系統設計 $\rightarrow$ 程式編碼 $\rightarrow$ 軟體測試 $\rightarrow$ 部署維護。每個階段都必須有完整的文檔產出，且前一階段未完全結束，後一階段就不能開始。
    
- **優點：** 結構極其簡單明瞭，管理成本低；適合**需求高度明確、幾乎不會變更**的傳統專案。
    
- **致命缺點：** 太晚看到成果。通常要到開發末期（測試階段）才會看到真實軟體，一旦此時發現需求理解錯誤，修改的代價巨大，極不具備彈性。
    

## 2. V 模型 (V-Model)

V 模型是對瀑布模型的延伸，它將**測試活動提早到開發的初期**，強調「開發階段」與「測試階段」的一對一對稱關係。

- **核心機理（驗證與確認）：** * 左半邊下降是開發與設計（由宏觀到微觀）。
    
    - 右半邊上升是測試與驗收（由微觀到宏觀）。
        
    - 在寫「需求定義」時，就必須同步編寫「驗收測試計畫」；在做「詳細設計」時，同步編寫「單元測試案例」。
        
- **優點：** 大幅提高軟體品質，測試人員能及早介入。
    
- **適用情境：** 對**安全性與可靠性要求極高**的系統（如醫療設備、高鐵車載通訊系統）。
    

## 3. 生魚片模型 (Sashimi Model)

生魚片模型本質上是「帶有階段重疊的瀑布模型」，由日本科學家提出。因為重疊的階段在圖表上看起來像層層堆疊的生魚片而得名。

- **演進邏輯：** 瀑布模型要求前一階段 100% 完工才能動工下一階段，太過死板。
    
- **核心機制：** 允許相鄰的開發階段**適度重疊（Overlap）**。例如，當需求分析完成了 80% 且核心架構已定案時，系統設計人員就可以提前進場動工，不需要傻傻等待最終的需求說明書定稿。
    
- **優點：** 縮短整體的開發總週期，且階段間的資訊傳遞更流暢；但缺點是階段邊界模糊，專案進度管理難度提升。
    

## 4. 漸進式模型 (Incremental Model)

漸進式模型採取「先廣度、後深度」的戰略。它將整個系統拆分成數個可獨立交付的「增量（Increments）」，每一次循環都交付系統的一部分功能。

- **白話範例（開發 Microsoft Word）：**
    
    - **增量一：** 只做出「打字與文字儲存」功能（核心產品交付）。
        
    - **增量二：** 加入「排版與字體調整」功能。
        
    - **增量三：** 加入「拼字檢查與表格」功能。
        
- **優點：** 用戶可以極快拿到核心產品（Time to Market 极短），且每次增量都可以作為下一步的調整依據。
    

## 5. 疊代式模型 (Iterative Model)

疊代式模型採取「先粗糙、後精細」**的戰略。它與漸進式不同，它在一開始就交付**完整系統的初版雛形，隨後透過一次次疊代（Iterations）來優化、重構、修正這個雛形。

- **白話範例（畫一幅蒙娜麗莎）：**
    
    - **疊代一：** 用鉛筆畫出全圖的輪廓（雖然模糊，但看到了全貌）。
        
    - **疊代二：** 上大塊的背景色彩。
        
    - **疊代三：** 精細雕刻五官與衣服光影。
        
- **優點：** 從一開始就能評估全局架構的風險，適合架構複雜但細節尚未完全定案的專案。
    

## 6. 螺旋模型 (Spiral Model)

螺旋模型（由 Barry Boehm 提出）是將瀑布模型與原型法結合，並引入了極其嚴格的「風險分析（Risk Analysis）」的重型開發模型。

- **運作機制：** 開發過程圍繞著四個象限，像螺線一樣一圈一圈往外擴大：
    
    1. 制定目標與限制條件。
        
    2. **風險評估與緩解（核心）：** 評估技術可行性、資安風險、或市場接受度。若風險過高，甚至可能在此時中止專案。
        
    3. 實作工程（編碼、測試）。
        
    4. 規劃下一階段。
        
- **適用情境：** **規模極其巨大、高成本、且充滿不確定性與高風險的國防或敏感政府大型專案**。它的代價是需要配置極高昂的風險評估專家。
    

## 7. 統一製程模型 (Unified Process - UP)

統一製程模型（UP / RUP）是一個基於物件導向、**以架構為中心（Architecture-centric）**、用例驅動（Use-case driven）且反覆疊代的現代化重型軟體開發框架。

它將專案的生命週期劃分為四個由淺入深的**核心階段（Phases）**：

1. **初始階段 (Inception)：** 確立專案的業務可行性與邊界，定義高階需求，評估「自建或購買（Buy vs Build）」的戰略決策。
    
2. **細化階段 (Elaboration)：** **最重要的階段**。團隊必須在此階段分析領域知識、建立**系統的核心架構藍圖（Baseline Architecture）**，並消滅大部分的高技術風險。
    
3. **構建階段 (Construction)：** 大規模的程式碼編寫與軟體開發，利用模組化與封裝原則，將系統各個功能模組組裝、測試完畢。
    
4. **移交階段 (Transition)：** 進行部署規劃（Deployment Planning）、資料庫遷移、系統上線，並執行 Beta 測試與用戶驗收（Validation）。
    

## 💡 軟體製程總結與思維演進

從歷史的維度來看，軟體製程模型的演進是一部「與不確定性對抗」的歷史：

- **瀑布 / V 模型（預測型）：** 假設世界是完美的，需求在第一天就能完全定死。適合合規要求高、變更代價大的硬體整合或航太專案。
    
- **漸進 / 疊代 / UP（演進型）：** 承認人類無法在第一天看清所有細節，因此透過分塊（增量）或逐步求精（疊代）來降低架構崩潰的風險。
    
- **敏捷開發（響應型，如 Scrum）：** 進一步將疊代縮短到 2~4 週，成為目前網路、SaaS 軟體開發的主流。
    

選擇製程模型時，沒有最好的模型，只有最適合該專案商業背景、技術風險與團隊組織結構的製程。

### Agile Development
歡迎來到敏捷開發（Agile Development）與現代創新管理的核心世界。如果說傳統的瀑布模型或統一製程（UP）是規劃周密的「重裝步兵」，那麼敏捷與精益就是應對快速市場變化的「特種部隊」。

以下為您系統性地拆解敏捷的核心精神、兩大主流執行框架，以及如何與商業創新結合：

## 1. 敏捷宣言 (Agile Manifesto)

敏捷開發並不是一種具體的程式寫法，而是一套**價值觀與原則**。2001 年，17 位軟體工程大師在美國猶他州聚會，共同簽署了《敏捷宣言》，確立了 4 大核心價值：

> - **個人與互動** 重於 流程與工具
>     
> - **可用的軟體** 重於 全面的文件
>     
> - **客戶協同合作** 重於 合約協商
>     
> - **回應變化** 重重 遵循計畫
>     

- **核心誤區修正：** 敏捷**不是**不要文件、不要計畫或合約，而是當兩者衝突時，敏捷更看重前面的價值。它的終極目標是透過縮短反饋週期，快速交付真正對用戶有價值的產品。
    

## 2. Scrum 框架

Scrum 是目前最普及、結構最嚴謹的敏捷實踐框架。它將開發過程切分為一個個固定長度（通常為 1 到 4 週）的**衝刺（Sprint）**。

### Scrum 的三大角色 (Roles)：

- **產品負責人 (PO - Product Owner)：** 負責決定「要做什麼」。他管理並排定產品需求池（Product Backlog）的優先順序，確保團隊在做價值最高的事情。
    
- **Scrum Master (SM)：** 團隊的「教練」與「障礙清除者」。他不指派工作，而是負責確保 Scrum 流程順暢，保護團隊不受外部干擾。
    
- **開發團隊 (Developers)：** 跨職能的自組織團隊，包含工程師、測試員、設計師，負責在衝刺內將需求轉化為「可交付的軟體增量」。
    

### Scrum 的四大會議 (Events)：

1. **衝刺規劃會議 (Sprint Planning)：** 團隊從 Backlog 挑選高優先順序的工作，承諾本次衝刺要完成的目標。
    
2. **每日站會 (Daily Standup)：** 每天 15 分鐘，每人分享：昨天做了什麼、今天預計做什麼、遇到什麼阻礙。
    
3. **衝刺檢視會議 (Sprint Review)：** 衝刺結束時，向 PO 和利益相關者（Stakeholders）展示真正做出來的產品成果（Demo），獲取直接回饋（Validation）。
    
4. **衝刺自省會議 (Sprint Retrospective)：** 團隊內部檢討「流程」與「協作」，找出下一個衝刺可以優化的 1~2 個具體行動。
    

## 3. 看板管理 (Kanban)

與 Scrum 這種高度結構化、有固定週期的做法不同，Kanban（源自豐田汽車的精益生產）是一種**基於資料流、非同步且連續**的視覺化管理方法。

### 看板的核心原則：

- **視覺化工作流 (Visualize Workflow)：** 使用看板（如 Jira、Trello）將工作切分為「待辦（To Do）」、「開發中（In Progress）」、「測試中（Testing）」、「完成（Done）」等狀態。每張卡片代表一個任務，進度一目了然。
    
- **限制在製品數量 (WIP Limit - Work in Progress)：** **這是看板最重要的精髓**。例如，限制「開發中」的 WIP 上限為 3。如果已經有 3 件事在做了，團隊就**絕對不能**再拉新任務進來，必須合力把這 3 件事推向「測試」或「完成」。這能有效暴露系統瓶頸，避免工程師同時多工（Multi-tasking）導致效率低下。
    
- **管理流動 (Manage Flow)：** 不強調衝刺，而是專注於縮短「週期時間（Cycle Time）」，即一個任務從開始動工到完全交付所需的時間。
    

## 4. 精益創業 (Lean Startup)

精益創業（由 Eric Ries 提出）將敏捷開發的精神從「軟體工程」放大到了「商業戰略」層面。它專門解決新產品開發時最大的風險：**「花了兩年、幾百萬台幣，最後做出一款根本沒有人要用的產品。」**

它提倡 **「建構—衡量—學習（Build-Measure-Learn）」** 的快速反饋循環：

1. **最小可行性產品 (MVP - Minimum Viable Product)：** 不要等到產品完美才上線。用最快、最省錢的方式做一個包含核心功能（甚至只是個漂亮的網頁收集 Email）的 MVP。
    
2. **驗證性學習 (Validated Learning)：** 快速將 MVP 推向市場，收集真實用戶的行為數據（例如點擊率、留存率），而不是聽信盲目的市調。
    
3. **軸轉或堅持 (Pivot or Persevere)：** 根據數據結果，如果發現當前的假設是錯的，立刻果斷**軸轉（Pivot，調整商業模式或核心方向）**；如果對了，就加碼研發（Persevere）。
    

## 💡 總結：現代軟體與商務創新的終極組合

在現代高效能企業中，這四個概念往往是完美融合在一起的：

- 公司使用 **Lean Startup** 的思維，先用 MVP 去測試市場，確認商機與用戶真正的痛點。
    
- 一旦方向確定，研發團隊採用 **Scrum** 的節奏進行反覆疊代與開發，或者利用 **Kanban** 嚴格限制 WIP 數量、優化每天的開發產出流動。
    
- 整套架構的底層，全部依賴 **Agile Manifesto** 所倡導的彈性、客戶協同合作與隨時擁抱變化的開放心態。

### AI Development
歡迎來到軟體工程的最新演進前沿：**AI 輔助軟體開發（AI-Driven Software Engineering）**。這幾年大型語言模型（LLM）與 Agent 技術的爆發，徹底重塑了從需求、架構、編碼到測試的整個軟體開發生命週期（SDLC）。其中最有趣的現象，莫過於開發典範從過去的「手動敲代碼」演進到現代的「氛圍編程（Vibe Coding）」。

以下為您系統性地拆解 AI 在各個軟體工程階段的核心應用與最新變革：

## 1. AI 用於需求工程 (AI for Requirements)

在傳統需求工程中，最耗時的是將客戶模糊的自然語言，轉化為無歧義的規格書（SRS）。AI 在此階段扮演了「高級分析師」與「邏輯翻譯官」的角色。

- **核心應用：**
    
    - **需求梳理與翻譯：** AI 可以直接消化大量的客戶訪談逐字稿、會議紀錄，自動分類並提煉出**功能需求（FR）**與**非功能需求（NFR）**。
        
    - **規格書自動生成：** 輸入高階業務目標，AI 能自動生成符合 IEEE 800-1998 或 **WRSPM 模型** 標準的軟體需求規格書。
        
    - **矛盾與漏洞檢測：** AI 能進行靜態邏輯分析，指出需求中的潛在衝突（例如：「規格 A 要求系統 0.1 秒內回應，但規格 B 要求進行高複雜度加密，兩者在硬體限制下可能衝突」）。
        

## 2. AI 用於架構設計 (AI for Architecture Design)

系統架構決定了軟體的骨架。AI 雖然無法完全取代資深架構師的直覺，但能大幅加速設計模式的評估。

- **核心應用：**
    
    - **架構方案評估（Trade-off Analysis）：** 你可以給 AI 具體情境（例如：「我需要建立一個高併發、資料高度解耦的系統」），AI 會為你對比 **Pipe & Filter**、**Client-Server** 或**微服務架構**的優缺點，並給出決策矩陣。
        
    - **架構圖（Diagram as Code）生成：** AI 能根據結構描述，自動生成 **PlantUML** 或 **Mermaid** 程式碼，直接渲染出系統架構圖、資料庫 E-R 圖或序列圖。
        
    - **技術選型建議：** 評估開源元件與第三方 SaaS 服務，輔助進行「自建或購買（Buy vs Build）」的策略評估。
        

## 3. AI 輔助編碼 (AI Assisted Coding)

這是目前發展最成熟、落地最廣泛的領域（如 GitHub Copilot、Cursor、Claude 3.5 Sonnet 等工具）。AI 已經從簡單的「自動補全（Autocomplete）」演進為「上下文感知的結對程式設計師（Pair Programmer）」。

- **核心應用：**
    
    - **上下文程式庫感知（Repository-wide Context）：** 現代 AI 工具能直接閱讀整個專案目錄的所有源碼（包含 C++、Python 檔案與 CMake 配置），理解專案的編寫風格與模組依賴，給出精準的程式碼擴充建議。
        
    - **程式碼重構（Refactoring）：** 選取一段複雜冗長的程式碼，AI 能在幾秒內將其優化為符合 **高內聚、低耦合** 原則、更具可讀性的現代代碼結構。
        
    - **文檔與註解自動化：** 自動為複雜函數生成 Doxygen、Docstring 或 Readme 文件。
        

## 4. 氛圍編程 (Vibe Coding)

這是 2024 至 2026 年軟體工程界最火熱的新名詞。**Vibe Coding（氛圍編程）** 是由前特斯拉 AI 總監 Andrej Karpathy 等人推廣的概念。

- **什麼是 Vibe Coding？**
    
    - 它是指工程師**不再親手撰寫任何一行底層程式碼**（不碰語法、不對齊括號、不處理記憶體指針）。
        
    - 工程師的角色轉變為「系統導演 / 指揮官」。你的職責是調配好「氛圍」（設定精確的 Prompt、明確的系統邊界、完美的驗證邏輯），然後放手讓 AI Agent（如 Devin, Claude Engineer）去瘋狂查閱文件、編寫代碼、編譯並修復錯誤。
        
- **開發典範的轉移：**
    
    - 舊時代：程式設計（Programming） = 70% 語法控制與打字 + 30% 邏輯思考。
        
    - Vibe Coding 時代：軟體開發 = 10% 創意/需求 + 80% 審查/架構評估 + 10% 測試驗證。工程師的核心競爭力變成了「如何問出好問題（Prompting）」**與**「如何精準進行 Verification & Validation（驗證與確認）」。
        

## 5. AI 用於測試與除錯 (AI Testing & Debugging)

編碼變快了，意味著產生 Bug 的速度也變快了。AI 技術在此時成了最強大的品質守門員。

- **核心應用：**
    
    - **單元測試自動生成：** AI 能一鍵分析 C++ 或 Python 類別，自動生成覆蓋率極高的單元測試（Unit Tests）腳本，包含極端狀況（Edge Cases）與模擬物件（Mocks）的編寫。
        
    - **智慧日誌分析與 Debug：** 當生產環境拋出崩潰日誌（Stack Trace）時，AI 能在幾秒內定位出是哪一個模組的哪一行程式碼（例如指針懸空或陣列越界）引發的，並直接給出修復（Git Patch）。
        
    - **自動化黑箱測試生成：** AI 驅動的 Agent 可以像真實用戶一樣，在網頁或 APP 的 UI 上隨機點擊（Monkey Testing）或模擬業務流程，自動探索並找出系統漏洞。
        

## 💡 總結：AI 時代下的工程師新思維

進入 AI 驅動開發的時代，工程師並不會失業，但**平庸的「碼農（Code Monkey）」會被淘汰**。

未來的核心軟體工程師，其技能樹將更偏向於**高階的軟體設計、系統架構與安全合規管理**。當 AI 幫我們把編碼和測試的泥淖填平時，工程師才能真正騰出大腦，去專注解決現實世界中最複雜的業務邏輯與創新痛點。
